1 Fundamentos

¿Qué es un agente de IA?

Un agente es un modelo de lenguaje (LLM) colocado dentro de un bucle: en lugar de responder una sola vez, recibe un objetivo, decide acciones, las ejecuta usando herramientas, observa el resultado y repite hasta cumplir la meta. La diferencia clave frente a un chatbot no es el modelo, sino la autonomía sobre acciones.

💬

Chatbot

Entrada → respuesta. Un turno. No actúa sobre el mundo; solo genera texto.

🔧

LLM + tools

Puede llamar funciones (buscar, calcular), pero tú orquestas cada paso.

🤖

Agente

Recibe un objetivo y decide por sí mismo qué herramientas usar, en qué orden y cuándo terminar.

Las tres propiedades que definen "agéntico"

  • Autonomía dirigida a un objetivo — opera sobre una meta de alto nivel ("arregla este bug"), no instrucciones paso a paso.
  • Uso de herramientas (tool use) — puede ejecutar acciones reales: leer archivos, correr comandos, llamar APIs, escribir código.
  • Bucle iterativo con retroalimentación — observa el resultado de cada acción y ajusta el plan. Es lo que permite recuperarse de errores.
Idea central

El salto de "asistente" a "agente" es poner el LLM dentro de un while loop con acceso a herramientas y un criterio de parada. Casi todo lo demás —MCP, skills, subagentes— son formas de darle mejores herramientas, más contexto o más manos.

2 Fundamentos

El loop agéntico

Casi todos los agentes modernos implementan alguna variante del patrón ReAct (Reasoning + Acting): el modelo alterna entre pensar y actuar. Cada vuelta del bucle es un "turno del agente".

🎯 Objetivo
prompt + contexto
→
🧠 Razonar
planear acción
→
⚙️ Actuar
llamar tool
→
👁️ Observar
leer resultado
↺
🧠 Re-evaluar
¿terminé?

Qué pasa en cada vuelta

  • 1. Razonar — el modelo recibe el historial completo (objetivo + resultados previos) y genera el próximo paso. A veces "piensa" en voz alta (chain-of-thought / extended thinking).
  • 2. Actuar — emite una tool call: una petición estructurada (JSON) para ejecutar una herramienta concreta con argumentos.
  • 3. Observar — el runtime (no el modelo) ejecuta la herramienta y devuelve el resultado al contexto como un nuevo mensaje.
  • 4. Decidir — con la nueva información, el modelo decide si hace otra acción o si ya tiene la respuesta final y detiene el bucle.
# Pseudocódigo del bucle agéntico
mensajes = [system_prompt, objetivo_usuario]
while True:
    respuesta = llm(mensajes, tools=tools_disponibles)
    mensajes.append(respuesta)

    if respuesta.tiene_tool_calls:
        for call in respuesta.tool_calls:
            resultado = ejecutar(call.nombre, call.args)   # el runtime actúa
            mensajes.append(tool_result(resultado))
    else:
        return respuesta.texto   # sin más acciones → terminado
Por qué importa el bucle

La recuperación de errores emerge del bucle: si un comando falla, el modelo ve el mensaje de error en la siguiente vuelta y puede corregir. Sin bucle, un solo fallo arruina toda la tarea. Por eso los agentes pueden encadenar cientos de llamadas a herramientas sin intervención humana.

3 Fundamentos

Anatomía de un agente

Todo agente, sin importar el proveedor, se compone de las mismas piezas. Entender estas capas te permite leer cualquier herramienta nueva y saber qué hace.

🧠 Modelo (LLM)
El "cerebro". Razona, planea y decide acciones. Ej: Claude Opus, GPT-5.5, Gemini.
📜 System prompt
Las instrucciones permanentes: rol, reglas, estilo, qué herramientas tiene y cómo usarlas.
🪟 Context window
La memoria de trabajo. Todo lo que el modelo "ve" ahora: prompt, historial, resultados de tools, archivos. Es finita.
⚙️ Tools
Las "manos": funciones que el modelo puede invocar (leer/escribir archivos, bash, web, APIs vía MCP).
🔁 Orquestador / runtime
El código que corre el bucle, ejecuta las tools de verdad, gestiona memoria y aplica permisos.

La pieza más incomprendida: la ventana de contexto

El modelo no "recuerda" nada entre llamadas. En cada vuelta del bucle se le re-envía todo el contexto relevante. La ventana tiene un límite (p. ej. 200K tokens). Gestionar qué entra y qué sale de esa ventana —context engineering— es la habilidad central de construir buenos agentes.

🎯

Tokens

Unidad de texto. La ventana se mide en tokens, no en palabras (~0.75 palabras/token).

📉

Context rot

Cuanto más llena la ventana, peor razona el modelo. Menos es más.

🗜️

Compactación

Resumir el historial viejo para liberar espacio sin perder lo esencial.

📂

Retrieval

Traer solo el fragmento necesario (de archivos, docs, memoria) justo a tiempo.

4 Fundamentos

Tools & function calling

Las herramientas son lo que convierte un modelo de texto en un agente que actúa. Técnicamente, una tool es una función con un nombre, una descripción y un esquema de parámetros (normalmente JSON Schema). El modelo no ejecuta la función: emite una petición para ejecutarla, y el runtime la corre.

Cómo se le describe una tool al modelo

{
  "name": "read_file",
  "description": "Lee el contenido de un archivo del disco.",
  "input_schema": {
    "type": "object",
    "properties": {
      "path": { "type": "string", "description": "Ruta absoluta" }
    },
    "required": ["path"]
  }
}
Clave de diseño

La descripción de la tool es prompt engineering puro: el modelo decide cuándo usarla solo a partir de ese texto. Una buena descripción —clara, con cuándo usar y cuándo no— vale más que código perfecto. Esto es exactamente lo que hace que las Skills y los MCP servers funcionen.

→

corre la función
→

vuelve al contexto

5 Fundamentos

Context engineering

Si prompt engineering es escribir buenas instrucciones, context engineering es gestionar todo lo que entra en la ventana del modelo durante una tarea larga: instrucciones, archivos, resultados de tools, memoria e historial. Es la disciplina que separa a un agente que funciona de uno que se confunde.

Las cuatro tácticas fundamentales

✍️

Write — escribir fuera

Guardar info en archivos/memoria externa (un scratchpad, notas) en vez de mantenerlo todo en la ventana.

🎯

Select — seleccionar

Traer a la ventana solo lo relevante ahora (retrieval, RAG, leer el archivo correcto justo a tiempo).

🗜️

Compress — comprimir

Resumir el historial viejo o los resultados largos para liberar tokens sin perder lo esencial.

🧩

Isolate — aislar

Repartir el trabajo entre subagentes, cada uno con su propia ventana limpia y enfocada.

Context rot

Más contexto no es mejor. A medida que la ventana se llena, la atención del modelo se diluye y empieza a ignorar instrucciones o mezclar datos. Por eso el objetivo no es "meter todo", sino mantener la ventana pequeña, relevante y limpia. Esta es la razón técnica de existir de las Skills (carga progresiva) y los subagentes (ventanas separadas).

Archivos de memoria persistente

Como el modelo olvida entre sesiones, las herramientas usan archivos convencionales que se cargan al inicio: CLAUDE.md (Claude Code), AGENTS.md (Codex, OpenCode y estándar emergente), reglas de proyecto. Son la "memoria a largo plazo" del agente sobre tu proyecto: estilo de código, comandos, convenciones.

6 Protocolos

MCP — Model Context Protocol

MCP es un estándar abierto (creado por Anthropic, hoy adoptado por OpenAI, Google y la mayoría de herramientas) para conectar agentes con herramientas y datos externos. Su analogía habitual: "el USB-C de la IA" — un único conector estándar en vez de una integración a medida por cada app.

El problema que resuelve

Sin MCP, conectar M agentes con N herramientas requiere M×N integraciones a medida. Con MCP, cada herramienta expone un servidor MCP y cada agente habla un cliente MCP: M+N. Cualquier agente que hable MCP puede usar cualquier servidor MCP.

🤖 Host / Cliente
El agente: Claude Code, Codex, Cursor, OpenCode…
⇄
JSON-RPC
stdio / HTTP
🔌 Servidor MCP
Expone: GitHub, Slack, Postgres, Figma, tu API…

Qué expone un servidor MCP

  • Tools — funciones que el agente puede ejecutar (ej: create_issue, query_db). Es lo más usado.
  • Resources — datos que el agente puede leer (archivos, registros, documentos), referenciados por URI.
  • Prompts — plantillas de instrucciones reutilizables que el servidor ofrece al usuario.

Transportes

TransporteCuándoEjemplo
stdioServidor local, mismo equipoUn proceso local que lee tus archivos o tu base de datos
HTTP + SSE / streamableServidor remoto, en la nubeUn conector SaaS (Slack, Linear) alojado por el proveedor

Así se registra un servidor MCP en un cliente (formato típico estilo Claude Code / muchos hosts):

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    }
  }
}

Un servidor MCP que expone una tool, con el SDK de Python (FastMCP):

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("clima")

@mcp.tool()
def get_clima(ciudad: str) -> str:
    """Devuelve el clima actual de una ciudad."""
    return consultar_api(ciudad)

mcp.run()  # expone la tool por stdio
  • 1. Al arrancar, el cliente pregunta al servidor tools/list → recibe los esquemas.
  • 2. El cliente inyecta esas tools en el contexto del modelo.
  • 3. El modelo decide usar una → el cliente llama tools/call al servidor.
  • 4. El servidor ejecuta y devuelve el resultado → vuelve al contexto del modelo.
Por qué MCP lo cambió todo

Antes, cada empresa de IA construía su propio sistema de plugins incompatible. MCP estandarizó la capa de herramientas, así que un servidor MCP de Notion funciona igual en Claude Code, Codex o Cursor. Es la razón de que el ecosistema de agentes haya explotado tan rápido.

7 Protocolos

Skills (Agent Skills)

Una Skill es una carpeta con instrucciones, scripts y recursos que enseña a un agente a hacer una tarea especializada. Si MCP da herramientas nuevas, una Skill da conocimiento y procedimiento: "cómo crear un PowerPoint con nuestra plantilla", "cómo revisar un PR según nuestras reglas".

Estructura de una Skill

mi-skill/
├── SKILL.md          # obligatorio: instrucciones + metadatos (YAML)
├── scripts/
│   └── generar.py    # código que el agente puede ejecutar
├── reference/
│   └── api.md        # docs que carga solo si las necesita
└── assets/
    └── plantilla.pptx

El SKILL.md empieza con un frontmatter que el agente lee al inicio:

---
name: pptx
description: Crear y editar presentaciones .pptx. Usar cuando el
  usuario mencione slides, deck o presentación.
---
# Instrucciones detalladas que se cargan solo al activarse...

El truco clave: carga progresiva (progressive disclosure)

Una Skill puede tener miles de líneas, pero no llena la ventana. El agente solo carga al inicio el name + description (unas pocas palabras). Si la tarea coincide con la descripción, entonces carga el cuerpo del SKILL.md; y solo si lo necesita, abre los archivos de reference/. Tres niveles, cargados bajo demanda:

Nivel 1 · Metadatos
name + description (~pocos tokens). Siempre cargado. Sirve para que el modelo decida si la skill aplica.
Nivel 2 · SKILL.md
El procedimiento completo. Se carga solo cuando la tarea activa la skill.
Nivel 3 · Recursos
Scripts, docs de referencia, plantillas. Se abren uno a uno solo si hacen falta.
Skills vs MCP — no compiten

Se complementan. Un MCP server conecta al agente con un sistema externo (te da la tool query_db). Una Skill le enseña el procedimiento y criterio ("cómo redactar el reporte mensual a partir de esos datos, con nuestro formato"). Una skill puede incluso indicar al agente qué tools MCP usar y cómo.

Quién las soporta

El formato de Skills (carpeta + SKILL.md) nació en Claude / Claude Code y hoy es soportado también por Codex (Agent Skills) y otras herramientas. Al ser archivos de texto plano, son portables, versionables en git y compartibles entre equipos.

8 Protocolos

Subagentes & sistemas multi-agente

Un subagente es un agente que otro agente invoca como si fuera una herramienta. El agente principal (orquestador) delega una sub-tarea a un subagente con su propia ventana de contexto limpia, recibe solo el resultado final, y sigue. Es la táctica isolate del context engineering llevada a su extremo.

Por qué delegar a subagentes

  • Aislar contexto — una búsqueda exhaustiva puede leer 50 archivos. Si eso ocurre en la ventana principal, la contamina. Un subagente lo hace en su ventana y devuelve solo la conclusión.
  • Paralelizar — varios subagentes pueden trabajar a la vez en partes independientes (explorar el frontend mientras otro explora el backend).
  • Especializar — cada subagente puede tener su propio system prompt, herramientas y permisos (uno solo-lectura para investigar, otro con escritura para implementar).

El patrón orquestador–trabajador

🧭 Orquestador
Descompone la tarea, reparte, integra resultados
⇊
delega
(tool: spawn)
🛠️ Subagentes
Worker 1 · Worker 2 · Worker 3
(ventanas independientes)
# El orquestador delega en paralelo
resultados = paralelo([
    subagente("explorer", "Mapea cómo funciona la auth en /backend"),
    subagente("explorer", "Mapea el flujo de login en /frontend"),
])
# cada subagente leyó decenas de archivos en SU ventana;
# el orquestador solo recibe 2 resúmenes limpios.
plan = orquestador.sintetizar(resultados)

Cuándo NO usar subagentes

El coste oculto

Cada subagente arranca "en frío": no comparte el contexto que ya tienes. Re-derivar ese contexto cuesta tokens, tiempo y dinero. Para tareas simples o muy acopladas, un solo agente en un bucle es más eficiente. Los subagentes brillan en trabajo amplio y paralelizable (búsqueda masiva, exploración multi-módulo, verificación independiente), no en todo.

Topologías multi-agente

PatrónCómo funcionaBueno para
Single agent loopUn agente, un bucle, muchas toolsLa mayoría de tareas. Empieza siempre aquí.
Orquestador–trabajadorUno reparte, varios ejecutan, uno integraInvestigación amplia, tareas paralelizables
Pipeline / secuencialSalida de un agente = entrada del siguienteFlujos por etapas (investigar→escribir→revisar)
Verificador / críticoUn 2º agente revisa el trabajo del 1ºCalidad, reducir errores, auto-corrección
9 Herramientas

Claude Code

Agente de codificación de Anthropic que vive en la terminal (también con extensiones de IDE y SDK). Es un agente "puro": un bucle con acceso a tu sistema de archivos, bash, web y MCP, diseñado para tareas de ingeniería de software de principio a fin.

📁

Acceso real al repo

Lee, edita y crea archivos; corre comandos y tests; trabaja sobre tu código de verdad, no copia-pega.

🧠

CLAUDE.md

Memoria de proyecto: convenciones, comandos, arquitectura. Se carga automáticamente.

🧩

Skills, MCP, subagentes

Soporta los tres mecanismos: extender con MCP, especializar con Skills, delegar con subagentes (tool Task).

⚡

Slash commands & hooks

Comandos personalizados, hooks para automatizar y plugins que empaquetan todo lo anterior.

Conceptos propios

  • Slash commands — atajos como /init, /review, o los tuyos, que inyectan un prompt predefinido.
  • Subagentes — vía la herramienta Task, con tipos especializados (explorador, planificador, etc.) y su propia ventana.
  • Hooks — scripts que se disparan en eventos (antes de editar, al terminar) para validar o automatizar.
  • Plugins & marketplaces — paquetes instalables que agrupan skills, MCP servers y comandos.
  • Claude Agent SDK — para construir tus propios agentes sobre el mismo motor.
10 Herramientas

Codex (OpenAI)

El sistema de agentes de codificación de OpenAI, en 2026 unificado sobre GPT-5.5 / GPT-5.5 Pro, con superficies de terminal (CLI), IDE, web y nube que comparten un mismo modelo de ejecución. Diseñado para correr tareas largas de forma autónoma.

🔁

Autonomía extrema

Demostrado encadenando 1.000+ llamadas a herramientas en tareas reales sin intervención; se auto-verifica antes de entregar.

☁️

Entornos en la nube

Clona tu repo en un sandbox, escribe en varios archivos, corre tests, itera sobre fallos y abre un PR.

🧩

Agent Skills

Soporta Skills para alinear el trabajo (documentación, prototipos, code understanding) con los estándares de tu equipo.

🤖

Automations

Trabaja sin que se lo pidan: triage de issues, monitoreo de alertas, CI/CD y tareas rutinarias.

Worktrees & paralelismo

Con worktrees integrados y entornos en la nube, varios agentes Codex trabajan en paralelo entre proyectos. Usa AGENTS.md como archivo de memoria de proyecto (estándar que comparte con otras herramientas). Reportan ~4 millones de desarrolladores activos semanales en 2026.

11 Herramientas

OpenCode

Agente de codificación open source nacido para la terminal, también con app de escritorio y extensión de IDE. Su rasgo distintivo: es agnóstico del modelo — funciona con 75+ modelos (Claude, OpenAI, Gemini, locales) y puede correr el bucle agéntico 100% local con Ollama o LM Studio.

🔓

Open source & privado

No almacena tu código ni contexto. Apto para entornos sensibles; con modelos locales nada sale de tu máquina.

🧭

Tipos de agente

Build (todo habilitado), Plan (analiza sin tocar), Review (solo-lectura), Debug (investigación con bash).

🔬

Integración LSP

Ve el código vía Language Server Protocol: tipos reales, firmas, imports y diagnósticos del compilador en 20+ lenguajes.

🔌

Multi-sesión & MCP

Varios agentes en paralelo sobre el mismo proyecto; conecta GitHub, Postgres, Slack vía MCP.

El concepto de "tipos de agente"

OpenCode hace explícito algo que conviene entender: un mismo motor con distintos permisos y tools es un agente distinto. Un agente Plan sin escritura no puede romper nada; un Review solo-lectura es seguro para auditar. Restringir herramientas es una forma de seguridad y de enfoque.

13 Práctica

Patrones agénticos

Cuatro patrones de diseño cubren la mayoría de los agentes efectivos. Se combinan entre sí.

📝

Planning

El agente descompone la meta en pasos antes de actuar. Mejora tareas largas y reduce que se pierda a mitad de camino.

🔧

Tool use

El más básico y potente: dar al modelo herramientas para interactuar con el mundo real en vez de solo generar texto.

🪞

Reflection

El agente revisa su propio trabajo y lo corrige antes de entregar. A veces con un segundo agente como crítico.

👥

Multi-agent

Repartir el trabajo entre agentes especializados que colaboran. Útil cuando una sola ventana no alcanza.

Reglas prácticas (lo que funciona)

Empieza con el agente más simple posible ›
No añadas multi-agente, orquestadores ni grafos complejos hasta que un solo bucle con buenas tools demuestre no alcanzar. La complejidad cuesta latencia, tokens y fallos.
Las descripciones de tools son tu producto ›
El modelo decide qué hacer leyendo nombres y descripciones. Inviértele tiempo: di claramente qué hace cada tool, cuándo usarla y cuándo no. Esto aplica igual a Skills y MCP servers.
Protege la ventana de contexto ›
Mantén la ventana pequeña y relevante. Usa carga progresiva (Skills), retrieval bajo demanda y subagentes para aislar trabajo ruidoso. Más contexto ≠ mejor.
Diseña para la recuperación de errores ›
Devuelve mensajes de error claros y accionables al modelo. Un buen agente se auto-corrige si ve qué falló. Errores silenciosos o crípticos rompen el bucle.
Añade un paso de verificación ›
Para trabajo importante, cierra con verificación: correr tests, releer el diff, un subagente crítico. Es el patrón reflection y reduce errores drásticamente.
Pon límites y permisos ›
Define qué puede hacer el agente sin pedir permiso y qué requiere confirmación (borrar archivos, mover dinero, hacer deploy). Restringir tools por rol (como los tipos de agente de OpenCode) es seguridad real.
